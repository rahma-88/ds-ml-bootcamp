# Data Cleaning and Preprocessing Decisions

The dataset was cleaned and preprocessed to improve data quality, ensure consistency, and prepare it for machine learning analysis. Each preprocessing step was selected based on the characteristics of the variables and standard data science practices.

First, the **Price** column contained currency symbols (`$`) and commas, making it a text variable instead of a numeric one. These characters were removed, and the column was converted into a floating-point numeric data type. This conversion was necessary because numerical operations such as calculating statistics, detecting outliers, and scaling require numeric values.

Next, inconsistencies were corrected in the **Location** column. The misspelled category **"Subrb"** was replaced with **"Suburb"**, while invalid entries represented by **"??"** were treated as missing values. Standardizing categorical values prevents duplicate categories that actually represent the same location and improves the accuracy of subsequent encoding.

Missing values were handled differently depending on the type of variable. For the **Odometer_km** feature, missing values were replaced using the **median** rather than the mean because odometer readings are often positively skewed and may contain extreme values. The median is resistant to outliers and therefore provides a more reliable estimate of the typical mileage. In contrast, **Doors** and **Location** are categorical variables, so missing values were filled using the **mode**, which represents the most frequently occurring category. Using the mode preserves the existing distribution of categorical data.

Duplicate records were then removed to eliminate redundant observations. Keeping duplicate rows could bias statistical summaries and machine learning models by giving repeated records greater influence than unique observations.

Outliers in **Price** and **Odometer_km** were handled using the **Interquartile Range (IQR) method**. The lower and upper limits were calculated as **Q1 − 1.5 × IQR** and **Q3 + 1.5 × IQR**, respectively. Instead of deleting observations outside these limits, the values were **capped (clipped)** to the calculated boundaries. IQR capping was chosen because it reduces the influence of extreme values while preserving every record in the dataset. This approach is particularly useful when outliers may represent legitimate but unusually expensive vehicles or cars with exceptionally high mileage.

The **Location** variable was transformed into numerical form using **one-hot encoding**, creating a separate binary column for each location category. This encoding allows machine learning algorithms to process categorical information without assuming any ordinal relationship between the categories.

Several new features were engineered to provide additional useful information. The **CarAge** feature was created by subtracting the manufacturing year from the current year (2026). Vehicle age is generally a stronger predictor of price than the raw manufacturing year because it directly measures depreciation. The **Is_City** feature was derived from the one-hot encoded location variables to create a simple binary indicator showing whether a car is located in a city. This feature can help models capture pricing differences between urban and non-urban markets. Additionally, a **LogPrice** feature was generated using the natural logarithm transformation (`log1p`). Since vehicle prices are often right-skewed, the logarithmic transformation reduces skewness, stabilizes variance, and produces a distribution that is often more suitable for predictive modeling.

Finally, numerical features were standardized using **StandardScaler**, which transforms each feature to have a mean of zero and a standard deviation of one. Standardization ensures that variables measured on different scales contribute equally to machine learning algorithms. The original **Price** and **LogPrice** columns were intentionally excluded from scaling because they represent the target variable and its transformed version. Likewise, one-hot encoded location variables and the binary **Is_City** feature were excluded because binary indicators already have meaningful values of 0 and 1 and do not require standardization.

Overall, these preprocessing decisions produced a clean, consistent, and machine-learning-ready dataset while preserving important information and minimizing the effects of missing values, inconsistent entries, and extreme observations.
